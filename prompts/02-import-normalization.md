# Prompt 02: 数据导入与标准化

## 目标

实现销项发票、进项发票、银行流水的导入与标准化能力。

## 你要做的事

- 支持三类导入：销项、进项、流水
- 增加导入批次记录
- 做字段校验、格式校验、重复检测、幂等导入
- 做客商名称标准化和金额方向标准化
- 先做导入预览，再做确认入库
- 产出导入错误报告或错误明细

## 约束

- 不要把导入逻辑和核销逻辑混在一起
- 导入失败必须能定位到具体记录
- 重复导入不能产生重复业务数据
- 重复导入要区分“重复跳过”和“状态更新”
- 对缺失官方唯一号的数据要有“疑似重复待确认”机制

## 交付物

- 导入接口或导入页面
- 标准化逻辑
- 导入批次追踪
- 逐行导入判定结果
- 示例模板或样例文件说明

## 验收标准

- 三类数据都可导入
- 重复导入可幂等处理
- 能区分新增、更新、重复、疑似重复、异常
- 有错误时能定位问题行
- 标准化后的数据可直接进入匹配引擎

## 可直接使用的 Prompt

```md
请基于当前仓库和文档，完成“数据导入与标准化”模块。

先阅读：
- docs/prd.md
- docs/domain-model.md
- docs/solution-design.md
- docs/task-breakdown.md

实现要求：

1. 支持导入销项发票、进项发票、银行流水三类数据。
2. 引入 ImportedBatch 或等价模型记录每次导入的来源、时间、总行数、成功数、失败数、状态。
3. 做字段校验、日期校验、金额校验、必填校验。
4. 实现导入预览机制：上传后先解析并输出分类结果，用户确认后再正式入库。
5. 做幂等导入和重复检测，避免重复入账。
6. 发票优先使用唯一业务主键防重：
   - 纸票/普票可使用发票代码 + 发票号码
   - 数电票可使用 20 位发票号码
7. 银行流水优先使用银行交易流水号 / 网银凭证号 / 企银交易序号防重。
8. 当官方唯一号缺失时，生成数据指纹作为兜底防重依据。
9. 区分并落地以下导入判定结果：
   - 新增
   - 状态更新
   - 重复跳过
   - 疑似重复待人工确认
   - 异常
10. 做客商名称规范化和金额方向标准化。
11. 为后续自动匹配提供干净、统一的 Invoice / BankTransaction 数据。
12. 补充必要测试或样例数据。

注意：
- 导入模块只负责接收和标准化数据，不直接做核销结论。
- 错误结果需要可追踪，最好能导出或查看明细。
- 不要把“状态更新”误判成“重复跳过”。
- 对指纹命中的数据，不要简单粗暴自动丢弃，必要时进入人工确认。

输出要求：
- 直接修改代码
- 说明导入流程和标准化策略
- 说明唯一主键和数据指纹分别怎么设计
- 说明如何验证三类导入成功
```
