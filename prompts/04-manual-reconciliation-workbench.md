# Prompt 04: 人工核销工作台

## 目标

实现财务核心操作界面和配套后端能力，让人工处理异常场景成为标准化动作。

## 你要做的事

- 实现左发票右流水工作台
- 支持筛选、搜索、候选建议查看
- 支持一对一、多对一、一对多、部分核销
- 支持确认核销
- 支持结构化异常处理
- 支持补录线下核销记录
- 为每次动作生成核销单和审计日志

## 约束

- 任何人工核销动作都必须可追溯
- 不允许只改状态不留核销单
- 页面重点是效率，不要做花哨 UI

## 交付物

- 工作台页面
- 人工核销接口
- ReconciliationCase / ReconciliationLine 落地
- 审计日志落地
- 右侧上下文操作区
- 销项 / 进项两套异常处理选项

## 验收标准

- 财务用户能完成主流程
- A/B/C/D/E/F/G 类异常都能在工作台完成处理
- 每次操作都能查看历史记录

## 可直接使用的 Prompt

```md
请基于当前仓库实现“人工核销工作台”。

先阅读：
- docs/prd.md
- docs/domain-model.md
- docs/solution-design.md
- docs/task-breakdown.md

需要实现的核心能力：

1. 提供左侧发票、右侧流水的人工核销工作台。
2. 支持按客商、金额、状态、日期、业务方向筛选。
3. 支持一对一、多对一、一对多、部分核销。
4. 当左侧和右侧都选中时，在页面右侧显示上下文操作区，而不是底部动作条。
5. 右侧操作区至少包含：
   - 确认核销
   - 异常处理
6. 异常处理必须按模式展示结构化选项：
   - 销项应收展示 `SO-A` 到 `SO-H`
   - 进项应付展示 `PI-A` 到 `PI-H`
7. 支持录入线下核销记录，例如现金支付、个人垫付等。
8. 每次人工动作都要生成 ReconciliationCase 和 ReconciliationLine。
9. 所有动作都必须记录审计日志和备注。

产品要求：
- 页面以效率为先，重点是列表、筛选、勾选、对比、确认。
- 不要再使用泛化的“报异常”按钮，必须是结构化“异常处理”。
- 不要只更新发票/流水状态，必须保留完整核销单。
- 需要兼容自动匹配引擎产出的候选结果。

输出要求：
- 直接修改代码
- 说明工作台支持了哪些操作
- 说明数据流如何从候选匹配进入核销单
```
