# Fin Ops Platform 预研说明

## 1. 项目定位

`fin-ops-platform` 的第一切入点是 **银企对账与发票核销（Reconciliation）**，目标是把财务目前依赖 Excel、网银导出、税务系统导出的人肉对账流程，沉淀成一个可追踪、可审计、可闭环跟进的平台。

但这个项目不应被设计成“单点核销工具”，而应该从一开始就预留成为 **财务运营平台** 的能力，后续逐步扩展到：

- `OA 系统接入`：承接报销、付款申请、审批单、项目主数据、供应商/客户主数据
- `项目成本测算`：以项目为主线归集收入、成本、票据、流水、预付款、待开票/待补票事项

## 2. 为什么先做银企核销

银企核销是最适合作为切入口的原因：

- 它直接连接 “票” 和 “钱”，业务价值最明确
- 异常场景多，人工痛点最强，最容易验证产品价值
- 一旦建好核销台账、提醒、审计链路，后续接 OA 和项目维度时复用率很高
- 核销过程天然会沉淀供应商/客户、项目、审批单、付款性质等关键财务主数据

## 3. 产品北极星

平台长期目标不是“对账成功”，而是建立一套统一的财务业务闭环：

1. 外部单据和内部单据统一归档
2. 发票、银行流水、审批单、项目维度可以相互关联
3. 自动化处理标准场景，人工只处理少量异常场景
4. 所有异常都有后续台账、责任归属、预计完成日和提醒机制
5. 每一笔收入/支出最终都能落到项目、业务事项或经营分析口径

## 4. 当前阶段建议范围

第一阶段建议只做 **核销闭环最小可用版本（MVP）**，优先支持：

- 销项发票导入
- 进项发票导入
- 银行流水导入
- 自动匹配与建议匹配
- 人工核销工作台
- 部分核销、合并核销、预收/预付、缺票、线下补充核销
- 催款/催票/退款台账
- 差额核销、红字发票、内部抵扣等高阶异常
- 完整操作日志和审计记录

第一阶段不建议重投入的内容：

- 复杂权限体系
- 多租户
- 全量 BI 分析体系
- 深度 OA 双向联动
- 复杂项目损益计算引擎

## 5. 对未来蓝图的预留要求

虽然当前只先做核销，但设计上必须预留以下边界：

- `对接层独立`：OA、网银、税务导入都不应直接写死在核心核销逻辑里
- `主数据独立`：公司主体、客商、项目、部门、审批单要有稳定 ID
- `核销对象可扩展`：现在关联发票和流水，未来需要能关联 OA 单据、项目维度、合同维度
- `状态机显式化`：已核销、部分核销、待催款、待催票、待退款、内部抵扣等状态要可枚举、可追踪
- `审计链路完整`：任何人工判断都应保留操作人、时间、备注和依据

## 6. 建议的阶段目标

### Phase 1: Reconciliation MVP

建立可运行的导入、自动匹配、人工核销、异常跟进闭环。

### Phase 2: Reconciliation Plus

补齐高阶异常、提醒中心、基础看板、批量操作、模板配置、规则配置。

### Phase 3: OA Integration

接入 OA 的审批单、付款申请、报销单、项目主数据，形成 “单据-发票-流水” 三方关联。

### Phase 4: Project Costing

在已核销数据之上建立项目收入/成本归集、预付款/待摊/待补票追踪与项目成本测算。
