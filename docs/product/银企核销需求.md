## 财务“银企对账与发票核销”系统需求逻辑文档

### 一、 系统定位与核心目标

本系统旨在通过对比**业务单据（OA / 税务发票）与资金流水（网银记录）**，实现应收/应付账款的自动化与半自动化核销，彻底解决财务手工对账效率低、异常款项（少付、溢收、缺票等）难追踪的痛点，并建立完善的账期提醒机制。

### 二、 整体操作主流程

1. **数据接入 / 导入**：系统实时接入 OA 系统（或接入 MongoDB 中间库）同步审批单数据；财务定期导入外部税务系统下载的《销项发票表》和《进项发票表》，以及从各家网银导出的《银行流水明细》。
2. **系统初筛**：系统根据“对方公司名称”与“金额”进行自动匹配，将完全对应的数据自动平账。
3. **人工干预（核心）**：提供“左发票、右流水”的双屏对比界面，由财务人员手动处理未匹配的异常数据。
4. **闭环跟进**：对于处理后产生的新状态（如欠款、欠票），系统自动生成台账与代办提醒。

---

### 三、 销项板块逻辑（应收账款管理：我司开票，资金流入）

**核心原则**：以“收款流水”为核销基准，寻找对应的“销项发票”。

- **正常场景 (1对1完美匹配)**
    - **逻辑**：同一公司，发票金额 = 收款金额。
    - **处理**：状态变更为【已核销】，业务闭环。
- **异常A：少付款 (部分核销)**
    - **逻辑**：发票金额 > 收款金额（例：开票50，实收20）。
    - **处理**：系统核销20元，发票剩余待收30元，状态变更为【部分核销】。
    - **后续**：强制要求设置“预计尾款收款日”，进入【催款台账】。
- **异常B：多票一付 (合并核销)**
    - **逻辑**：同一公司多张发票金额之和 = 一笔/多笔收款金额（例：开票25+30，实收55）。
    - **处理**：人工勾选多张发票与对应流水，点击“合并核销”，状态变更为【已核销】。
- **异常C：无票收款 (预收款 / 往来款)**
    - **逻辑**：存在收款流水，但无对应销项发票（例：收到100元，未开票）。
    - **处理**：人工将流水标记为特定性质（如：预收账款、押金、员工还款等）。若为需退还的款项，需设置“计划退款日”进入提醒；若为预收，则留存等待未来开票后抵扣。
- **异常F：溢收款 (超额核销)**
    - **逻辑**：收款金额 > 发票金额（例：开票30，实收50）。
    - **处理**：优先核销该发票30元（状态变更为【已核销】），剩余20元流水系统自动剥离，转化为上述的**异常C（预收/往来款）**由人工确认。

---

### 四、 进项板块逻辑（应付账款管理：别人开票，资金流出）

**核心原则**：以“付款流水”为核销基准，寻找对应的“进项发票”。

- **正常场景 (1对1完美匹配)**
    - **逻辑**：同一公司，发票金额 = 付款金额。
    - **处理**：状态变更为【已核销】，业务闭环。
- **异常D：多付款 / 缺票 (预付账款)**
    - **逻辑**：付款金额 > 发票金额（例：开票5，实付10）。
    - **处理**：核销5元发票，流水剩余5元未平账，标记为【缺票】。
    - **后续**：进入【催票台账】，跟进对方补开发票。
- **异常E：少付款 / 组合支付**
    - **逻辑**：发票金额 > 公账付款金额（例：开票10，公账仅付8）。
    - **处理**：人工介入，补充“线下核销记录”（如备注：剩余2元已通过现金支付/个人垫付等）。补充后，发票强制平账，状态变更为【已核销】。
- **异常G：一付多票 (合并核销)**
    - **逻辑**：一笔付款金额 = 同一公司多张进项发票金额之和。
    - **处理**：人工勾选单笔流水与多张发票，进行“合并核销”。

### 五、 全局通用与高阶逻辑

这部分处理现实财务中必然遇到的“非标准化”阻力。

- **异常H：银行手续费与抹零**
    - **逻辑**：因银行扣费或零头抹除，导致流水与发票存在微小差额（例：发票100，收款99）。
    - **处理**：提供【差额核销】功能。人工确认后，将差额（1元）标记为“财务费用/手续费/抹零”。发票视为【已核销】，不再产生催款提醒。
- **异常I：红字发票与退款 (负数冲销)**
    - **逻辑**：发生退货，税务开具负数发票（如-10元），同时网银产生反向流水（如付款退给客户10元）。
    - **处理**：系统必须支持负数金额的核销抵扣，允许销项界面的负数发票与付款流水进行平账。
- **异常J：债务互抵 (客商身份重叠)**
    - **逻辑**：某公司既是客户也是供应商，双方同意货款互抵（例：欠我司50，我司欠他30，最终只收回20）。
    - **处理**：支持“跨板块平账”。财务可通过人工录入内部抵扣单，将30元的销项发票与30元的进项发票直接内部核销，不关联实际银行流水。

### 六、关联工作台与税金抵扣新增需求（2026-03-25）

本章节用于补充 UI 工作台、字段展示层次以及税金抵扣入口的产品需求。

#### 6.1 主工作台入口与总体结构

- 主页面名称为：`OA & 银行流水 & 进销项发票关联台`。
- 页面继续采用**上下两栏**结构：

  - 下半区：`未配对`

- 页面风格应保持**Excel / 子表式工作台**，以表格、表头、行状态、行内操作为主，不采用大块卡片式布局。

#### 6.2 税金抵扣功能入口

- 在主工作台右上角新增按钮：`税金抵扣`。

- 主工作台顶部新增一个**日历 / 月份选择按钮**。
- 用户选择月份后，系统应按该月份刷新显示：
  - OA 数据
  - 银行流水
  - 进项发票
  - 销项发票
- 税金抵扣页面也应继承同一月份上下文，或允许在页面内再次切换月份。

#### 6.4 三栏拖拽与动态显示

- 三栏之间的两个竖向分隔线必须支持**拖拽调整宽度**。
- 拖拽后，三栏宽度需同步作用于：
  - 上半区“已配对”
  - 下半区“未配对”
- 分隔线需要支持**完整拖动**：
  - 可将某一栏拖至 0 宽或接近 0 宽，视为该栏被收起
  - 可从收起状态再次拖回，恢复该栏显示
- 页面应支持动态显示：
  - 一栏
  - 两栏
  - 三栏
- 当用户收缩某一栏到极小宽度时，可视为该栏被临时折叠；重新展开后恢复表格视图。
- 各栏内部允许独立滚动与横向滚动，保证字段较多时仍保持 Excel 式查看体验。

#### 6.4.1 关联台读写分离与性能要求（2026-04-08）

- 关联台后续不再采用“每次动作后重建整张关联台”的方式。
- 系统需要拆成两层：
  - 写模型：只处理 `确认关联 / 取消配对 / 忽略 / 异常处理` 等动作
  - 读模型：专门存储已经组装好的关联台快照
- 其中：
  - `pair relations` 只负责保存已配对关系
  - `row overrides` 只负责忽略、异常、备注等非配对覆盖
  - `workbench read model` 负责保存最终展示给前端的 `已配对 / 未配对 / 已忽略 / 已处理异常`
- 目标是：
  - `确认关联 / 取消配对` 后端快速返回
  - 前端成功后立即局部更新
  - 整页刷新优先读取缓存好的关联台快照
  - 必要时再后台静默刷新做校准
- 关联台缓存至少要支持两类作用域：
  - `all`
  - `YYYY-MM`

#### 6.4.2 三栏局部搜索 / 筛选 / 排序（2026-04-08）

- `已配对` 与 `未配对` 两个区域内，各自的 `OA / 银行流水 / 进销项发票` 三栏，需要支持独立的局部搜索、列筛选和时间排序。
- 该功能不是全局搜索，而是**当前区域内、当前栏驱动、整组联动**：
  - 当前栏只显示命中的项
  - 其他两栏显示与这些命中项属于同一候选组的全部相关项
- 每个区域内：
  - 一次只允许打开一个栏级搜索框
  - 切换到别栏时，前一个搜索框自动关闭
- 每栏标题后增加一个搜索 icon 按钮：
  - 点击后弹出一个小搜索框
  - 无需确认按钮
  - 用户输入时实时过滤结果
- 每个列名右侧增加一个下拉筛选按钮：
  - 多选
  - 支持全选 / 清空
  - 筛选生效后，同样按“当前栏驱动、整组联动”显示
- 排序规则：
  - `银行流水` 支持按时间升序 / 降序切换
  - `进销项发票` 也支持按时间升序 / 降序切换
  - 排序按“组”进行，使用当前栏记录时间作为该组排序键，保证三栏仍然按组对齐
- 字段裁剪要求：
  - `OA` 不再显示 `金额`、`申请事由`
  - `银行流水` 不再显示 `备注`
  - `进销项发票` 不再显示 `金额/税率/税额`、`价税合计`

#### 6.4.3 三栏列拖拽排序与登录持久化（2026-04-08）

- `已配对` 与 `未配对` 两个区域内，各自的 `OA / 银行流水 / 进销项发票` 三栏，需要支持列头拖拽排序。
- 拖拽规则：
  - 只在当前栏内部重排，不跨栏拖动
  - 拖动后列顺序立即生效
  - 表头、内容行、底部横向滚动轨道都必须保持同一列顺序与同一列宽口径
- 保存规则：
  - 列顺序需要保存为用户级配置
  - 用户下次登录后，系统按上次保存的列顺序恢复显示
  - `OA / 银行流水 / 进销项发票` 三栏分别独立保存
- 宽度规则：
  - 某一栏被整体拉宽或缩窄时，栏内所有列按比例联动变化
  - 不允许出现表头和内容区列边线错位、或列左右出现白色割裂带

#### 6.4.4 设置页数据重置与 OA 模式 B 重刷（2026-04-14）

- 在 `关联台设置` 中新增一个危险操作分组：`数据重置`。
- 该分组只对管理员可见，不对普通全量账号或只读导出账号开放。
- 分组内新增三个按钮：
  - `清除所有银行流水数据`
  - `清除所有发票（进销）数据`
  - `清除所有 OA 数据并重新写入`
- 三个按钮都必须有：
  - 明确影响说明
  - 二次确认
  - 当前 OA 用户密码输入弹窗
  - 执行中状态
  - 成功 / 失败反馈
- 用户点击任一清理按钮后，必须先完成影响范围确认，再输入当前登录 OA 账户的 OA 系统密码。
- 后端必须校验该密码属于当前 OA 会话用户，校验通过后才可进行清理；未输入、取消或校验失败时，不允许删除任何数据，也不允许触发 OA 重建。
- OA 密码只允许作为本次高风险动作复核使用，不允许保存、写日志、写审计明文、回显到错误信息或出现在 API 响应中。
- 这组按钮只允许清理 app 内部数据，不允许直接操作 OA 源库。
- 明确禁止触碰：
  - `form_data_db.form_data`
  - 全局设置
  - 内部元数据表
- 若 OA 密码复核服务本身报错，前端只能展示通用失败提示，不得把上游异常原文直接回显给用户，避免密码或敏感上下文泄露。
- `清除所有银行流水数据` 的业务语义：
  - 清空已导入银行流水
  - 清空相关匹配结果
  - 清空相关已配对 / 异常 / 忽略状态
  - 清空相关工作台缓存
- `清除所有发票（进销）数据` 的业务语义：
  - 清空已导入进项票 / 销项票
  - 清空税金认证导入结果
  - 清空相关匹配结果
  - 清空相关已配对 / 异常 / 忽略状态
  - 清空相关工作台缓存
- `清除所有 OA 数据并重新写入` 固定采用 **模式 B：彻底重刷 OA 相关状态**，具体含义是：
  - 清空 OA 附件发票缓存
  - 清空 workbench read model
  - 清空与 OA 相关的已配对关系
  - 清空与 OA 相关的异常 / 忽略覆盖
  - 再按设置中的 `保OA` 日期重新从 OA 源数据读取并构建
- 这里的“重新写入”不是删除 OA 原始数据再导入，而是：
  - 清 app 侧缓存和派生状态
  - 再按当前规则重建 app 侧可见结果
- 验收口径：
  - 三类动作必须有自动化测试覆盖清理边界与保护目标
  - `清除所有 OA 数据并重新写入` 必须验证按设置中的 `保OA` 日期重建
  - 管理员可见、普通全量账号不可见、只读导出账号不可见
  - 密码未输入 / 输错 / 上游校验失败时，不清任何集合、不触发 OA 重建、不回显 OA 密码

#### 6.13 OA 集成后的访问账户与操作权限模型（2026-04-07）

- 当前 app 接入 OA 后，账户状态必须先分成两类：
  - `在 OA 系统看得见并可访问此 app`
  - `在 OA 系统看不见且访问不了此 app`
- 对于“在 OA 系统看得见并可访问此 app”的账户，还需要继续分成两种操作权限：
  - `所有操作均可`
  - `只可看和只可导出`
- `所有操作均可` 的账户，允许使用当前 app 的全部能力，包括但不限于：
  - 导入
  - 确认关联 / 取消关联
  - 异常处理
  - 忽略 / 撤回忽略
  - 税金抵扣计划调整
  - 设置项维护
- `只可看和只可导出` 的账户，只允许：
  - 查看关联台、税金抵扣、成本统计
  - 使用搜索
  - 使用导出
  - 不允许做任何会改变业务状态的动作
- 当前只有 OA 账户 `YNSYLP005` 可以管理上述两层权限。
- `YNSYLP005` 在关联台 `设置` 中应新增访问账户管理能力，用于维护：
  - 哪些 OA 账户“在 OA 系统看得见并可访问此 app”
  - 这些可访问账户属于 `所有操作均可` 还是 `只可看和只可导出`
- 对于“不在可访问列表中的账户”，要求同时满足：
  - OA 菜单中看不见此 app
  - 直接输入 app 地址也无法访问
  - 所有 API 均返回无权限
  - 有效抵扣税额
  - 勾选状态
  - 发票状态
- 只有明确属于**已认证 / 已勾选 / 可用于抵扣**的进项票，才进入税金抵扣页面中的 `已认证结果`。
- 已认证导入完成后，系统必须按以下规则回写到 `进项票认证计划`：
  - 如果导入结果匹配到计划中的某一条发票：
    - 该计划票显示 `已认证`
    - 该行置灰
    - checkbox disabled
    - 自动纳入试算
  - 如果导入结果未匹配到当前计划：
    - 进入右侧抽屉 `已认证但未进入计划`
    - 仍然自动纳入试算
- 税金抵扣页面中的 `已认证结果` 后续必须来自真实导入与持久化结果，不再允许仅靠写死样例数据展示。
- 税金抵扣页面继续坚持：
  - 只做计划
  - 只做计算
  - 只做展示
- 税金抵扣页面不做：
  - 税局真实提交
  - 税局真实撤销
  - 真实申报工作流

#### 6.14 关联台确认关联 / 取消配对轻量写模型（2026-04-08）

- 当前关联台中的 `确认关联` 和 `取消配对` 不应再依赖“重建整张关联台 + 重写整份 override 快照”的重路径。
- 需要新增独立的 `pair relations` 持久化层，专门存储“哪些记录已组成一个配对组”。
- `pair relations` 只负责配对关系，不负责忽略、异常处理、备注覆盖等其他状态。
- `忽略 / 撤回忽略 / 异常处理 / 访问账户管理 / 设置类覆盖` 继续使用现有 settings / overrides 持久化体系，不混入 `pair relations`。
- `确认关联` 的目标行为：
  - 只写入一条或一组 `pair relation`
  - 后端成功后立刻返回
  - 前端立即把当前记录从 `未配对` 挪到 `已配对`
- `取消配对` 的目标行为：
  - 只删除或失效化当前 `pair relation`
  - 后端成功后立刻返回
  - 前端立即把当前组从 `已配对` 挪回 `未配对`
- `pair relations` 至少需要记录：
  - `case_id`
  - `row_ids`
  - `row_types`
  - `status`
  - `relation_mode`
  - `created_by`
  - `created_at`
  - `updated_at`
- 推荐 `status` 至少支持：
  - `active`
  - `cancelled`
- 推荐 `relation_mode` 至少支持：
  - `manual_confirmed`
  - `auto_internal_transfer`
  - `auto_salary`
- 关联台加载时，系统应先读取原始 OA / 银行流水 / 发票，再叠加 `active pair relations` 生成 `已配对 / 未配对` 分组。
- 自动匹配产生的工资、内部往来款等已配对关系，也应逐步统一落到 `pair relations`，避免“自动匹配规则”和“人工确认结果”分散在多套状态源里。
- 重构完成后，`确认关联 / 取消配对` 的主要耗时不应来自整表重算或整份 override 快照写入。

### 七、成本统计页面新增需求（2026-04-01）

本章节用于补充“成本统计”页面的业务需求。该页面建立在现有 `OA ↔ 银行流水 ↔ 发票` 工作台与项目归集底座之上，但定位是**统计与下钻页面**，不是新的核销弹窗。

#### 7.1 页面入口与位置

- 在顶部导航中，`税金抵扣` 右侧新增按钮：`成本统计`。
- 点击后进入一个**独立页面**，不是弹出窗口。
- 页面默认继承当前全局 `年份 / 月份` 上下文，也允许用户在页面内继续点击切换年月。

#### 7.2 成本统计数据来源

- 成本统计以**银行流水**为主数据源。
- 仅统计能够对应到 OA 字段的**支出类流水**。
- 每条进入成本统计的数据，至少需要能够取得以下 OA 字段：
  - 项目名称
  - 费用类型
  - 费用内容
- 如果一条流水尚未关联到 OA，或者 OA 中缺少上述关键字段，则该流水**不进入正式成本统计结果**。
- 收款流水、未确认关联流水、仅发票无流水的记录，不进入成本统计主视图。

#### 7.3 页面主视图一：按月份统计

- 页面默认打开 `按月份统计` 视图。
- 统计维度为：**当前所选月份**。
- 主表需要显示：
  - 项目名称
  - 费用类型
  - 金额
  - 费用内容
- 统计口径：
  - 在当前月份内，对已确认纳入成本统计的支出流水进行汇总展示。
  - 一行代表一组可解释的成本统计结果；允许按 `项目名称 + 费用类型 + 费用内容` 组合展示。
- 每一行均可点击继续下钻。

#### 7.4 页面主视图二：按项目名称统计

- 页面需支持按 `项目名称` 查看成本。
- 用户可从 `按月份统计` 中点击某一项目下钻进入，也可直接切换到 `按项目统计` 视图。
- 该视图需要显示：
  - 时间
  - 费用类别 / 费用类型
  - 金额
  - 费用内容
- 该视图仍然处于当前月份范围内，除非后续单独扩展跨月统计。

#### 7.5 层层下钻要求

- 成本统计页面中的每一条都应支持继续点击下钻。
- 推荐下钻链路为：
  1. 月份统计汇总
  2. 项目名称维度明细
  3. 具体银行流水明细
- 最终下钻到某一条具体流水时，页面需要能看到：
  - 该条银行流水本身信息
  - 对应 OA 的项目名称
  - 对应 OA 的费用类型
  - 对应 OA 的费用内容
- 下钻过程应保持在同一成本统计页面内，通过面包屑、返回按钮或层级切换完成，不使用弹窗承载主流程。

#### 7.6 导出要求

- 成本统计页面必须提供 `导出` 功能。
- 导出应至少支持导出**当前视图**的数据：
  - 当前月份汇总视图
  - 当前项目明细视图
  - 当前下钻到的流水明细视图
- 导出文件名应带有清晰上下文，例如：
  - 年月
  - 当前视图层级
  - 项目名称（如适用）

#### 7.7 页面排版要求

- 页面排版必须清晰，不可做成拥挤的纯技术报表。
- 推荐保留：
  - 年月选择
  - 统计视图切换
  - 面包屑 / 返回路径
  - 汇总信息区
  - 主表区
  - 导出按钮
- 主表仍以清晰的表格为主，保证财务人员可以快速：
  - 看项目
  - 看费用类型
  - 看金额
  - 看费用内容
  - 继续下钻

#### 7.8 OA 费用类型字典

成本统计页面中，`费用类型` 采用以下 OA 字典值：

- 设备货款及材料费
- 人工费/劳务费/服务费
- 住宿费
- 招待费（餐费、烟酒等）
- 交通费
- 车辆使用费（汽油、过路、保险、维修、税费等）车辆维修
- 运费/邮费/杂费
- 房屋使用费（户租、水电、维修、车位、屋业等）
- 经营/办公费用
- 财务费用
- 借款
- 还款
- 其他
- 固定资产

#### 7.9 与现有页面的关系

- `关联台` 负责核销、异常处理、忽略、已处理异常等流程。
- `税金抵扣` 负责进销项税额抵扣。
- `成本统计` 负责基于已纳入统计口径的银行流水做项目与费用视角统计。
- 三个页面共用同一套年份 / 月份上下文，但职责不同：
  - 关联台：处理关系
  - 税金抵扣：处理税额
  - 成本统计：处理费用统计与下钻分析
