# 路线图

## 1. 总体策略

采用 “先核销闭环，后系统连接，再成本测算” 的三段式路线：

1. 先解决财务最强痛点
2. 再把财务业务连接到 OA 和主数据体系
3. 最后把数据沉淀为项目成本能力

## 2. 分阶段规划

## Phase 0: 基础搭建

目标：

- 建立仓库结构、基础数据模型、导入批次模型、审计日志基础能力

产出：

- 技术骨架
- 核心实体定义
- 枚举和状态机基础
- 导入样例和测试样例

## Phase 1: 银企核销 MVP

目标：

- 让财务可以导入数据、自动匹配、人工核销、查看异常台账

核心交付：

- 发票导入
- 流水导入
- 自动匹配
- 人工核销工作台
- 催款/催票/退款台账
- 审计日志

验收标准：

- 标准场景可自动核销
- 异常场景可人工闭环
- 未闭环事项自动生成台账

## Phase 2: 核销增强版

目标：

- 提升复杂场景覆盖率和财务处理效率

核心交付：

- 差额核销
- 红字发票处理
- 内部抵扣
- 批量操作
- 提醒中心
- 匹配规则配置化

验收标准：

- A-J 异常场景全覆盖
- 复杂人工场景有明确审计链路

## Phase 3: OA 集成

目标：

- 把审批流与财务核销打通，形成“业务单据-发票-流水”的统一视图

核心交付：

- OA 项目主数据同步
- OA 报销单/付款申请单同步
- 单据与发票/流水关联
- 集成适配层和失败重试机制

验收标准：

- 可以从核销记录追溯到 OA 单据
- 可以按 OA 项目或审批单查看核销情况

## Phase 4: 项目成本测算

目标：

- 基于已打通的数据链路建立项目成本归集和测算能力

核心交付：

- 项目维度挂接
- 成本归集规则
- 预付款/缺票/待摊事项的项目归属
- 项目收入成本看板

验收标准：

- 项目维度可查看收入、成本、未闭环事项
- 可支持基础项目毛利和成本偏差分析

## 3. 关键里程碑

### M1

导入与标准化跑通

### M2

自动匹配与人工核销工作台可用

### M3

台账闭环与提醒可用

### M4

高阶异常全覆盖

### M5

OA 集成底座可用

### M6

项目成本测算可试运行

## 4. 风险与控制点

- 银行模板和税务导出格式可能频繁变化
- 客商名称不规范会显著影响自动匹配效果
- 如果没有独立核销单模型，复杂异常会很快失控
- 如果项目维度不提前预留，后续成本测算会变成补历史数据工程

## 5. 当前建议优先级

当前最优先的仍然是：

1. 导入与标准化
2. 匹配引擎
3. 人工核销工作台
4. 台账与提醒
5. 高阶异常
6. OA 适配层预留
7. 项目维度预留
