# 项目状态管理与成本统计范围设计

日期：2026-04-14

## 目标

在设置页新增可用的 `项目状态管理` 能力，并让成本统计页面根据该设置决定统计范围。

本轮需要完成三件事：

- 设置页可以管理项目，并把项目分成 `进行中` 和 `已完成`
- 项目可以从 OA 项目管理中拉取，也可以由用户在 app 内手动新增或删除
- 成本统计页面在 `按时间 / 按项目 / 按银行 / 按费用类型` 右侧增加范围 trigger，可切换 `进行中` 和 `所有项目`

默认统计范围为 `进行中`。这样日常看成本时不会被已完成项目干扰，但用户仍能一键切回 `所有项目` 做历史复核。

## 业务口径

### 项目状态

项目只有两个业务状态：

- `进行中`
- `已完成`

设置页的 `项目状态管理` 是成本统计过滤项目状态的唯一来源。成本统计页面不再自行推断项目是否完成。

### 项目来源

项目来源分两类：

- OA 项目管理拉取的项目
- app 内手动新增的项目

OA 项目管理只读接入。app 不允许修改或删除 OA 源系统里的项目数据，也不得写入 `form_data_db`。

### 删除项目

删除项目只作用于 app 本地项目状态管理，不删除 OA 源项目，也不删除历史成本、OA、发票或银行流水。

建议规则：

- 手动新增项目可以从本地项目管理中删除
- OA 拉取项目如果允许删除，只能删除本地状态覆盖或从当前管理列表隐藏
- 下一次 OA 项目同步如果再次拉到同一项目，该项目可以重新出现在设置页

删除项目不是“从成本统计永久排除项目”。如果后续需要“排除项目”功能，应另建明确字段，避免和删除项目混在一起。

### 未登记项目

成本统计中遇到未出现在设置页项目列表里的项目时，默认按 `进行中` 处理。

原因：

- 避免新 OA 项目或历史数据因为设置未同步而从成本统计中消失
- 用户明确把项目标记为 `已完成` 后，才从默认 `进行中` 范围中排除

## 当前系统基础

当前后端设置服务已经有项目状态雏形：

- `AppSettingsService.get_settings_payload()` 返回 `projects.active`、`projects.completed`、`projects.completed_project_ids`
- `AppSettingsService.update_settings()` 已能保存 `completed_project_ids`
- 设置页已有 `项目状态管理` 分组和 `标记完成 / 移回进行中` 基础交互

当前缺口：

- 设置页还不能稳定从 OA 项目管理主动拉取项目
- 设置页还缺少完整的新增 / 删除项目交互
- 成本统计服务还没有基于项目状态的 `进行中 / 所有项目` 范围过滤
- 成本统计 API 和前端请求还没有项目范围参数

## 数据模型建议

继续复用现有 app settings，增量扩展项目管理字段。

建议后端在 settings snapshot 中维护：

- `completed_project_ids`: 已完成项目 ID 列表，继续保留
- `manual_projects`: app 内手动新增的项目列表
- `project_overrides`: 可选，存储 OA 项目的本地覆盖信息，例如本地状态、隐藏标记或显示名修正

项目唯一标识建议：

- 优先使用 OA 项目 ID 或现有 `ProjectMaster.id`
- 手动项目使用稳定本地 ID，例如 `proj_manual_xxxx`
- 如果成本统计只有项目名称，没有项目 ID，则用规范化项目名称作为 fallback key

状态判断建议封装成服务方法，而不是散落在页面或统计函数里：

- `get_project_status(project_id, project_name) -> active | completed`
- `is_project_active(project_id, project_name) -> bool`
- `list_active_project_keys()`

这样成本统计、导出和下钻可以共用同一个判断。

## 后端设计

### 设置服务

`AppSettingsService` 需要成为项目状态管理的后端入口：

- 返回 OA 项目 + 手动项目的合并列表
- 返回每个项目的 `id / project_code / project_name / project_status / source / department_name / owner_name`
- 支持把项目标记为 `进行中` 或 `已完成`
- 支持新增手动项目
- 支持删除本地项目或本地覆盖
- 支持触发 OA 项目同步

OA 同步必须走现有集成服务或项目归集服务，不能直接写 OA 数据库，更不能改 `form_data_db`。

### 成本统计服务

成本统计服务增加 `project_scope` 参数：

- `active`: 只统计 `进行中` 项目
- `all`: 统计所有项目

默认值为 `active`。

过滤应尽量在构建成本条目后、汇总前集中处理，保证以下接口口径一致：

- 月份汇总
- explorer
- 项目视图
- 银行视图
- 费用类型视图
- 导出预览
- 导出文件
- 项目下钻
- 交易详情中与成本列表相关的上下文

如果某条成本记录没有项目 ID 但有项目名称，使用项目名称 fallback 判断。无法识别的项目默认视为 `进行中`。

### API 合同

成本统计相关 GET 接口增加 query 参数：

- `project_scope=active`
- `project_scope=all`

未传时按 `active` 处理。非法值应返回 400，或安全回落到 `active` 并在测试中固定口径；建议返回 400，避免用户误以为已切换成功。

设置相关接口可采用两种方式实现：

- 在现有 `/api/workbench/settings` 的 `GET / PUT` payload 中扩展项目字段
- 新增更细的项目管理 action endpoint，例如 `POST /api/workbench/settings/projects/sync`、`POST /api/workbench/settings/projects`、`DELETE /api/workbench/settings/projects/{project_id}`

建议优先采用小 endpoint，降低一次性保存整个 settings payload 时误覆盖其他设置的风险。

## 前端设计

### 设置页

设置页 `项目状态管理` 保持树状两栏风格，不引入花哨卡片。

建议结构：

- 顶部工具区：`从 OA 拉取项目`、`新增项目`
- 左栏：`进行中`
- 右栏：`已完成`
- 每个项目行展示项目名称、项目编码、来源
- 行内操作：`标记完成`、`移回进行中`、`删除`

删除前需要确认，文案明确说明不会删除 OA 源项目和历史数据。

### 成本统计页

在 `按时间 / 按项目 / 按银行 / 按费用类型` 右侧增加范围 trigger：

- 默认显示 `进行中`
- 点击可切换到 `所有项目`
- 再次点击或选择可切回 `进行中`

trigger 风格要和现有成本统计 toolbar 一致，不要新增明显突兀的视觉系统。

切换范围后需要重新拉取成本统计数据，并把范围参数带到：

- explorer 请求
- 月份汇总请求
- 项目详情请求
- 导出预览请求
- 导出请求

## 不做范围

本轮不做：

- 不修改 OA 源系统项目状态
- 不删除 `form_data_db` 或 `form_data_db.form_data`
- 不清理历史 OA、发票、银行流水或 pair 数据
- 不新增“排除项目”语义
- 不重构成本统计页面整体布局
- 不改变成本统计既有入账口径，只在项目状态维度增加范围过滤

## 验收标准

- 设置页能看到 `项目状态管理`
- 用户能从 OA 项目管理拉取项目
- 用户能新增本地项目
- 用户能把项目在 `进行中 / 已完成` 之间切换
- 用户能删除本地项目或本地项目状态覆盖，且不会影响 OA 源项目
- 成本统计页面默认只展示 `进行中`
- 成本统计页面可切换到 `所有项目`
- 成本统计的列表、汇总、下钻、导出预览和导出使用同一个项目范围
- 未登记项目默认按 `进行中` 处理
- 所有实现和 prompt 均明确禁止改动 `form_data_db`
