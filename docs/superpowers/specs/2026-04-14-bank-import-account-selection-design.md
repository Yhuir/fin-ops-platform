# 银行流水导入按账户映射项主导设计文档

日期：2026-04-14

## 目标

让银行流水导入时“用户上传时选择的银行账户映射项”成为该文件识别和入库的主依据，而不是只作为前端模板猜测。

这里的“账户映射项”指设置页 `银行账户映射` 中的单条记录，语义为：

- `mapping_id`
- `bank_name`
- `last4`

## 需求结论

本轮采用最小改造方案：

- 银行流水导入弹窗中，用户为每个文件选择的是“具体账户项（银行名 + 后四位）”，不是只有银行名
- 前端预览请求把 `mapping_id / bank_name / last4` 一起传给后端
- 后端预览和确认导入时，都以“所选账户项”为主来源
- 自动识别继续保留，但只做校验，不再主导最终标签
- 如果自动识别结果和所选账户项不一致，在预览结果页先提示冲突，确认导入前弹确认框
- 用户确认后，仍按所选账户项导入

## 现状问题

当前链路存在三个问题：

1. 上传弹窗里虽然要求用户选银行，但前端只把它当作模板代码猜测依据。
2. 后端标准化后的银行标签和后四位标签，仍然主要依赖文件内容中的账号和设置里的 `last4 -> bank_name` 映射。
3. 当用户明确知道文件属于哪个账户时，系统没有把这个信息贯穿到导入结果中，导致“用户选择”和“最终标签”可能不一致。

这会直接影响：

- 银行流水显示标签
- 后四位 tag
- 后续成本统计和关联台对账户维度的理解

## 设计原则

### 1. 用户选择优先

对银行流水导入来说，用户在上传时选择的账户项是业务输入，不是提示信息。  
本轮把它定义成主依据。

### 2. 自动识别只负责发现冲突

保留现有自动识别能力，因为它仍然能发现：

- 文件模板是否选错
- 文件内容里的账号后四位是否和用户选择不一致
- 文件内容里的银行特征是否和用户选择不一致

但自动识别不再覆盖用户选择。

### 3. 只影响新导入，不改历史数据

本轮不回刷历史已导入流水，不改历史银行标签，也不改历史后四位 tag。  
所有新规则只对新上传文件生效。

## 方案概览

### 前端上传选择

银行流水导入弹窗的数据源改成设置页的 `bank_account_mappings` 完整列表。  
每个下拉选项显示：

- `银行名 + 后四位`

例如：

- `建设银行 8826`
- `工商银行 1045`

前端不再把选项先压平成“去重后的银行名”。

### 预览请求载荷

银行流水文件的预览 override 从：

- `bank_name`

升级成：

- `bank_mapping_id`
- `bank_name`
- `last4`

后端收到后，把这组值挂到该文件 preview item 和 normalized row 上下文里。

### 后端标准化来源

对于银行流水文件，标准化结果中的账户标签来源顺序改成：

1. 用户选择的账户项
2. 文件内容自动识别结果，仅用于冲突检测和补充诊断
3. 现有兜底逻辑，仅用于无法识别时生成提示，不再改写主标签

也就是说：

- 银行标签按用户选择的 `bank_name`
- 后四位 tag 按用户选择的 `last4`

### 冲突检测

预览阶段新增银行选择冲突判断。

冲突来源包括：

- 文件内容中解析出的账号后四位，与所选 `last4` 不一致
- 文件模板或账号前缀推断出的银行，与所选 `bank_name` 不一致

预览结果里新增文件级冲突字段，例如：

- `bank_selection_conflict: true`
- `detected_bank_name`
- `detected_last4`
- `conflict_message`

### 确认导入前弹窗

前端在用户点击“确认导入”时，如果 preview 中存在冲突文件，则先弹确认框。

弹框语义：

- 系统识别结果是什么
- 用户当前选择的账户项是什么
- 是否仍按用户选择的账户项导入

如果用户取消：

- 不执行确认导入

如果用户继续：

- 按用户选择的账户项入库

## 数据结构调整

### 前端类型

银行流水导入文件 override 增加：

- `bankMappingId?: string | null`
- `bankName?: string | null`
- `last4?: string | null`

预览结果文件项增加：

- `selectedBankMappingId?: string | null`
- `selectedBankName?: string | null`
- `selectedBankLast4?: string | null`
- `bankSelectionConflict?: boolean`
- `detectedBankName?: string | null`
- `detectedLast4?: string | null`
- `conflictMessage?: string | null`

### 后端会话模型

`UploadedImportFile` 和 `FileImportPreviewItem` 需要同时保存：

- 用户所选账户项
- 自动识别账户结果
- 冲突状态

这样预览和重试时不会丢失上下文。

## 设置页边界

本轮不改设置页结构。  
继续复用现有 `bank_account_mappings`：

- `id`
- `last4`
- `bank_name`

上传弹窗只需要从现有设置 payload 读取完整列表，不再做按银行名去重。

## 接口行为调整

### `POST /imports/files/preview`

银行流水文件的 override 需要接受：

- `bank_mapping_id`
- `bank_name`
- `last4`

响应中的每个文件项需要返回：

- 所选账户项
- 自动识别结果
- 是否冲突
- 冲突文案

### `POST /imports/files/confirm`

默认行为不变。  
新增一个前提：

- 如果存在冲突文件，前端必须先走冲突确认后再调用 confirm

本轮推荐保持 confirm 接口本身简单，不新增复杂状态机。冲突确认先由前端 gate。

## 识别口径

### 银行标签

最终写入和展示使用用户选择的 `bank_name`。

### 后四位 tag

最终写入和展示使用用户选择的 `last4`。

### 自动识别结果

仅用于：

- 冲突判断
- 预览诊断展示
- 后续问题排查

不作为最终写入依据。

## 风险与处理

### 1. 同一文件内容和用户选择明显冲突

处理：

- 预览中显式标红提示
- 确认前强制弹窗
- 用户决定是否继续

### 2. 设置里同一银行存在多个账户项

处理：

- 上传时要求选择具体账户项
- 不再允许只选银行名

### 3. 模板识别和账户选择绑得太紧

处理：

- 模板识别仍独立运行
- 账户选择只影响最终银行标签和后四位 tag
- 不把模板判断完全绑定到账户项

## 不做内容

本轮不做：

- 历史导入流水重刷
- 设置页银行账户映射结构重构
- 多级冲突处理流程
- 服务端记忆“用户曾确认过某类冲突”
- 其它导入类型的账户项选择

## 验收标准

- 银行流水导入时，每个文件必须选择具体账户项才能预览
- 预览请求能把 `mapping_id / bank_name / last4` 传到后端
- 导入后的银行标签和后四位 tag 以用户所选账户项为准
- 自动识别与用户选择冲突时，预览结果可见冲突信息
- 点击确认导入时，存在冲突文件会先弹确认框
- 用户确认继续后，文件按所选账户项导入
- 不影响发票导入和历史已导入数据
