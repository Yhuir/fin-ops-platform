# 关联台强搜索设计

日期：2026-04-02

## 目标

为关联台提供一个真正可用的强搜索能力，支持跨 `OA / 银行流水 / 发票` 的统一搜索、筛选、分组展示与跳转定位。

这个搜索不是简单的“当前表格筛一遍”，而是一个独立的搜索入口，支持：

- 全局关键词搜索
- 按月份搜索
- 按项目搜索
- 按人名搜索
- 按发票号搜索
- 按流水号搜索
- 按公司名搜索
- 按状态搜索
- 从搜索结果一键跳回关联台并定位到目标记录

## 背景

当前系统里：

- 关联台有 `已配对 / 未配对 / 已忽略 / 已处理异常`
- 成本统计本质上复用了 `OA + 银行流水`
- 税金抵扣本质上围绕 `发票`

但还没有一个统一的检索入口。用户想查某个对象时，往往只能：

- 手动切月份
- 在页面里肉眼找
- 在不同页面来回切

这对真实业务场景不够用，尤其是下面这些高频查找：

- 找某个申请人提交过的 OA
- 找某个项目相关的付款流水
- 找某家公司相关的票、流、OA
- 找某张发票现在在什么状态
- 找某笔银行流水对应了什么 OA

## 范围

本轮设计覆盖：

- 关联台顶部搜索入口
- 搜索弹窗交互
- 统一搜索接口设计
- `OA / 流水 / 发票` 三组结果模型
- 跳转回关联台并定位的规则
- 搜索字段与筛选器
- 第一版实现边界

本轮不做：

- 导入记录搜索
- 成本统计独立结果页搜索
- 税金抵扣独立搜索页
- 拼音首字母检索
- 自然语言查询
- 搜索历史 / 收藏搜索
- Elasticsearch 级全文引擎

## 设计原则

### 1. 搜索按“实体”统一，不按“页面”割裂

搜索结果按底层实体返回：

- OA
- 银行流水
- 发票

不因为用户当前在哪个页面，就采用不同口径。

### 2. 搜索入口简单，结果能力强

页面上只保留一个清晰搜索入口；复杂度放在弹窗内，而不是把页面头部做成筛选器大杂烩。

### 3. 搜到只是第一步，能跳到才有价值

每条结果都必须可跳转。用户需要的不只是“搜出来”，而是：

- 自动切到对应月份
- 回到关联台
- 滚动到记录
- 高亮记录

### 4. 结果按实体分组，不做三栏候选布局

搜索结果弹窗里不再复用关联台的三栏布局，而是普通结果列表，降低认知负担。

### 5. 第一版先把强筛选和定位做稳

先把字段覆盖、月份 / 项目 / 状态筛选、结果分组、跳转定位做好；更重的模糊拼音、搜索历史、语义搜索留到后续。

## 关联台顶部结构调整

当前右上区域里：

- 删除 `关联台月份 2026-03` 这块纯文案
- 保留页内月份选择器
- 在顶部工具栏新增一个宽搜索框

搜索框文案建议：

搜索框行为：

## 搜索弹窗

### 整体结构

弹窗分三块：

1. 顶部搜索区
2. 中部筛选区
3. 下部结果区

### 顶部搜索区

包含：

- 一个主搜索输入框
- 搜索范围切换：
  - `全部`
  - `OA`
  - `流水`
  - `发票`

搜索输入支持回车搜索，也支持输入后短延迟自动搜索。

### 中部筛选区

第一版建议支持以下筛选：

- 月份：
  - `全部月份`
  - `单月`
- 项目：
  - `全部项目`
  - 某个项目
- 状态：
  - `全部状态`
  - `已配对`
  - `未配对`
  - `已忽略`
  - `已处理异常`

### 下部结果区

按三组展示：

- OA
- 银行流水
- 发票

每组：

- 有独立标题
- 显示命中数量
- 支持折叠 / 展开（第一版可默认展开）

- 金额

## 发票可搜索字段

- 购方名称
- 销方税号
- 购方税号
- 金额
- 发票类型
- 月份

## 额外建议支持的强搜索

除了用户已经明确提出的内容，第一版建议额外支持：

- 金额搜索
  - 如：`58000`
- 账户后四位搜索
  - 如：`8106`
- 日期 / 月份搜索
  - 如：`2026-03`
  - 如：`2026-03-28`
- 状态搜索
  - 如：`已配对`
  - 如：`已忽略`

## 搜索归一化规则

第一版建议做弱归一化，而不是重型全文检索：

- 忽略前后空格
- 中英文括号弱化
- 公司名常见后缀弱化匹配：
  - `有限公司`
  - `有限责任公司`
  - `股份有限公司`
- 大小写不敏感
- 银行卡尾号允许纯数字命中
- 发票号 / 流水号支持完整匹配与包含匹配

## 搜索结果项设计

## OA 结果项

展示结构：

- 主标题：项目名称
- 第一辅助行：申请人 | 对方户名 | 金额
- 第二辅助行：费用类型 | 费用内容
- 右侧标签：月份 / 状态 / 命中字段
- 动作：`跳转至`

## 银行流水结果项

展示结构：

- 主标题：对方户名 / 公司名
- 第一辅助行：交易时间 | 金额 | 收入/支出
- 第二辅助行：支付账户 | 流水号 / 核心流水号
- 第三辅助行：摘要 / 备注
- 右侧标签：月份 / 状态 / 命中字段
- 动作：`跳转至`

## 发票结果项

展示结构：

- 主标题：发票号码 / 数电发票号码
- 第一辅助行：销方名称 | 购方名称 | 金额
- 第二辅助行：开票日期 | 发票类型
- 右侧标签：月份 / 状态 / 命中字段
- 动作：`跳转至`

## 命中高亮

结果项里只高亮真正命中的片段，不整行染色。

同时增加“命中字段”标签，例如：

- `命中：项目名称`
- `命中：对方户名`
- `命中：流水号`
- `命中：发票号码`

这样用户能理解这条为什么被搜出来。

## 跳转定位设计

搜索结果只提供一个主动作：`跳转至`

点击后执行：

1. 关闭搜索弹窗
2. 切换关联台月份到结果所属月份
3. 加载该月份工作台数据
4. 判断目标记录所属区域：
   - 已配对
   - 未配对
   - 已忽略
   - 已处理异常
5. 自动展开 / 打开对应区域或对应弹窗
6. 滚动到目标记录
7. 给目标记录加一段短时高亮

### 跳转区域规则

- `已配对`
  - 展开 `已配对` 区
- `未配对`
  - 展开 `未配对` 区
- `已忽略`
  - 先打开 `已忽略` 弹窗，再滚动定位
- `已处理异常`
  - 先打开 `已处理异常` 弹窗，再滚动定位

### 跳转统一落点

无论用户当前在哪个页面发起搜索，`跳转至` 一律回到关联台定位。

原因：

- 搜索是全局入口
- 关联台才是处理 OA / 流水 / 发票关系的主工作面

## 后端接口设计

推荐新增统一搜索接口：

- `GET /api/search`

参数建议：

- `q`
- `scope=all|oa|bank|invoice`
- `month=YYYY-MM|all`
- `project_name`
- `status`
- `limit`

返回结构建议：

```json
{
  "query": "建设银行 8106",
  "filters": {
    "scope": "all",
    "month": "all",
    "project_name": null,
    "status": null
  },
  "summary": {
    "total": 12,
    "oa": 3,
    "bank": 5,
    "invoice": 4
  },
  "oa_results": [],
  "bank_results": [],
  "invoice_results": []
}
```

### 单条结果字段建议

每条结果至少包含：

- `row_id`
- `record_type`
- `month`
- `zone_hint`
- `case_id`
- `matched_field`
- `matched_text`
- `title`
- `primary_meta`
- `secondary_meta`
- `status_code`
- `status_label`
- `jump_target`

其中 `jump_target` 用于前端直接执行定位，不需要自己猜。

## 后端实现建议

第一版不建议引入 Elasticsearch。

建议复用现有的：

- OA Mongo 读取能力
- app Mongo 里的银行流水 / 发票 / 状态明细
- workbench 分组与 override 能力

做一个统一 `SearchService`，负责：

- 读取底层实体
- 做字段归一化
- 应用筛选条件
- 计算结果所属区域
- 生成统一结果模型

## 性能与索引建议

第一版建议在 Mongo 明细 collection 上补这些索引：

- `bank_transactions.month`
- `bank_transactions.counterparty_name`
- `bank_transactions.transaction_id`
- `bank_transactions.account_last4`
- `invoices.month`
- `invoices.invoice_no`
- `invoices.digital_invoice_no`
- `invoices.counterparty_name`
- `workbench_row_overrides.row_id`
- `workbench_row_overrides.status_code`

对 OA 只读库：

- 通过现有月份过滤 + 项目名 / 申请人弱过滤收口
- 不直接改写 OA 库结构

第一版上线时，至少满足：

- 可以按关键词搜索 `OA / 流水 / 发票`
- 可以按月份筛选
- 可以按项目筛选
- 可以按状态筛选
- 结果按 `OA / 流水 / 发票` 三组展示
- 每条结果能看到命中字段
- 点击 `跳转至` 后：
  - 能切月份

  - 能定位并高亮记录

  - 已配对
  - 未配对

## 推荐实施顺序

1. 先做后端统一搜索接口
2. 再做关联台顶部搜索入口和搜索弹窗
3. 再做结果跳转与定位
4. 最后补高亮、筛选和回归测试

## 结论

推荐方案是：

- 关联台顶部一个统一搜索入口
- 点击后打开搜索弹窗
- 结果按 `OA / 流水 / 发票` 分组展示
- 提供月份 / 项目 / 状态筛选
- 每条结果都支持 `跳转至`
- 跳转统一落到关联台并完成月份切换与定位高亮

这套方案兼顾了：

- 搜索能力强
- 页面入口简洁
- 实体口径统一
- 和现有工作台结构兼容
