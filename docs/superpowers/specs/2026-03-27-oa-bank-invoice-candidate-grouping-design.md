# OA-流水-发票高置信自动闭环与候选组工作台设计

日期：2026-03-27

## 目标

把当前“银行流水 ↔ 发票自动匹配、OA 主要靠人工确认”的工作台，升级成：

- 只对高置信场景做自动闭环
- 其余全部落到人工匹配区
- `已配对` / `未配对` 两个区域都按“候选组”横向展示
- 让财务能在同一水平线直观看到 `OA | 银行流水 | 发票` 三栏候选关系

## 问题背景

当前工作台的核心问题有两个：

1. 自动匹配边界不清晰  
   现在真正自动跑的是 `银行流水 ↔ 发票`，而 `OA ↔ 流水 ↔ 发票` 三方并没有形成统一的自动链路。OA 更多是同步进工作台后等待人工确认。

2. 页面缺少“同组候选横向对齐”  
   当前 `OA / 银行流水 / 发票` 仍然是三张独立表，各自按自己的顺序往下排。即使几条记录实际上属于同一候选，也只能靠用户目测和选中联动去理解，核销效率低。

## 范围

本轮覆盖：

- 重新定义自动闭环边界，只保留高置信自动闭环
- 为 `OA / 银行流水 / 发票` 增加统一候选分组模型
- 调整 `/api/workbench` 返回结构，从“按栏平铺 rows”升级成“按组返回 groups”
- React 工作台改成“每组一行，三栏横向对齐”的渲染模式
- `已配对` / `未配对` 两个区域都使用相同的候选组布局

本轮不做：

- 全量复杂语义自动核销
- 手续费 / 利息 / 电信托收 / 内部往来等复杂异常的自动闭环
- 强语义模糊户名识别
- 超过三笔以上的组合自动求和闭环
- 多步骤审批流或新的财务台账模型

## 核心原则

### 1. 自动闭环必须保守

自动闭环的设计目标不是“多匹配”，而是“少误判”。

系统只对唯一、无歧义、关系明确的记录自动闭环。只要存在：

- 多候选
- 金额差异
- 户名只相似不相同
- 需要依赖摘要语义判断
- 需要借助业务背景解释

就不自动闭环，只进入候选组等待人工处理。

### 2. 人工工作台必须以“候选组”为中心

页面不再优先强调“某一栏有多少条记录”，而是优先强调“这几条是否构成同一组候选”。

因此，渲染粒度从“单条记录”切换成“候选组”：

- 一组 = 一行
- 一行内包含：
  - `OA` 单元格
  - `银行流水` 单元格
  - `进销项发票` 单元格

缺失的栏位保留空格，不向上或向左补齐。

## 自动闭环边界

### 允许自动闭环的场景

#### A. `OA ↔ 银行流水`

满足以下全部条件时，允许自动闭环：

- 对方户名归一后完全一致
- 金额完全一致
- 付款 / 收款方向一致
- 日期在近窗口内
- 候选唯一

#### B. `银行流水 ↔ 发票`

满足以下全部条件时，允许自动闭环：

- 对方户名归一后完全一致
- 金额完全一致
- 方向一致
- 候选唯一

#### C. `OA ↔ 银行流水 ↔ 发票`

满足以下全部条件时，允许自动闭环：

- 三者能构成唯一链路
- 不存在第二组等价候选
- 金额没有差额
- 方向不冲突

### 不允许自动闭环、只进入候选组的场景

- 一个 OA 多个流水
- 多个 OA 一笔流水
- 一票多收 / 多票一付
- 金额不完全一致但接近
- 户名只是相似，不是完全一致
- 手续费、利息、贷款、电信托收
- 借款、还款、预付、退款
- 内部往来
- 任何依赖摘要、备注或业务背景才能判断的场景

## 候选组模型

### 新增工作台候选组概念

工作台的基础数据结构从：

- `paired.oa[]`
- `paired.bank[]`
- `paired.invoice[]`
- `open.oa[]`
- `open.bank[]`
- `open.invoice[]`

升级为：

- `paired.groups[]`
- `open.groups[]`

每个 `group` 表示一组横向对齐候选。

### Group 字段

建议字段：

- `group_id`
- `group_type`
  - `auto_closed`
  - `manual_confirmed`
  - `candidate`
- `match_confidence`
  - `high`
  - `medium`
  - `low`
- `reason`
- `oa_rows[]`
- `bank_rows[]`
- `invoice_rows[]`

### 分组优先级

#### 1. 优先按正式 `case_id`

只要记录已经有正式 `case_id`，就直接归到同一候选组。

这适用于：

- 自动闭环结果
- 人工确认后的结果
- 已有核销案例回灌

#### 2. 没有 `case_id` 的，用临时候选键归组

临时候选键建议由以下因素组成：

- 业务方向
- 归一化对方户名
- 金额主值
- 日期窗口

第一版不追求“绝对最优分组”，只追求：

- 分得稳
- 可解释
- 不把明显不相关的记录硬塞到一起

### 金额不一致时的分组策略

金额不一致但有可能是同一业务时：

- 可以进入同一候选组
- 但 `group_type` 仍然是 `candidate`
- 状态需要明确标成待人工确认

也就是说，“能同组展示”不等于“能自动闭环”。

## 页面布局

### 区域结构

工作台仍保留两个区域：

- `已配对`
- `未配对`

但区域内部不再是三张独立表，而是一个“候选组列表”。

### 每组一行三栏

每一个候选组渲染为一行：

- 左：OA
- 中：银行流水
- 右：进销项发票

每栏都可能出现：

- 0 条
- 1 条
- 多条

当某栏无数据时：

- 当前单元格留空
- 其他栏不补位
- 下一组从下一行开始

这样可以保证用户始终沿一条水平线观察同组候选。

### 栏内多条记录

如果同一组某栏有多条记录：

- 单元格内部采用垂直堆叠
- 每条仍保留选中、详情、确认关联、标记异常等动作
- 但这些动作仍然作用于“记录级”，不是整个组强绑定批量提交

## 自动与人工区的页面口径

### 已配对

只显示：

- 高置信自动闭环结果
- 人工确认后的结果

不显示仍待判断的候选组。

### 未配对

显示：

- 全部待人工处理候选组
- 包括：
  - 缺 OA
  - 缺流水
  - 缺发票
  - 有多候选
  - 有金额差异

## 后端改造

### 1. 增加 Candidate Grouping 层

在现有：

- `MatchingEngineService`
- `WorkbenchQueryService`
- `LiveWorkbenchService`

之间增加一层候选组装配逻辑，职责是：

- 读取 OA、银行流水、发票
- 先识别高置信自动闭环
- 再把剩余记录按临时候选键装配成候选组
- 输出适合工作台展示的数据结构

### 2. `/api/workbench` 契约升级

现有接口需要从“分栏数组”升级为“按组返回”。

建议形态：

```json
{
  "month": "2026-03",
  "summary": {
    "oa_count": 59,
    "bank_count": 10,
    "invoice_count": 10,
    "paired_count": 4,
    "open_count": 75,
    "exception_count": 12
  },
  "paired": {
    "groups": [
      {
        "group_id": "grp_001",
        "group_type": "auto_closed",
        "match_confidence": "high",
        "reason": "exact_oa_bank_invoice_match",
        "oa_rows": [],
        "bank_rows": [],
        "invoice_rows": []
      }
    ]
  },
  "open": {
    "groups": []
  }
}
```

### 3. 自动闭环逻辑收口

自动闭环不再只关注 `流水 ↔ 发票`，而是要允许：

- 先构建 `OA ↔ 流水` 高置信对
- 再构建 `流水 ↔ 发票` 高置信对
- 最后合并成唯一三方链路

但每一层都必须遵守“候选唯一”的规则。

## 前端改造

### 1. 从 Pane Rows 升级成 Group Rows

当前前端类型以：

- `paired.oa`
- `paired.bank`
- `paired.invoice`

为核心。

本轮需要升级成：

- `paired.groups`
- `open.groups`

并增加候选组映射类型。

### 2. WorkbenchZone 改成按组渲染

当前 `WorkbenchZone` 和 `ResizableTriPane` 以“三栏独立表”渲染。

本轮要改成：

- 三栏仍可拖拽宽度
- 但主体改成“按组选行”
- 一行内部有三格
- 每格内部渲染当前栏的 0~N 条记录

### 3. 详情与操作保留记录级

即便页面按组渲染：

- `详情`
- `确认关联`
- `标记异常`
- `取消关联`

仍然是记录级动作，不直接把整个 group 绑定成单一动作对象。

## 错误处理

### 1. 分组失败兜底

如果某条记录无法可靠分组：

- 允许单独形成一个 group
- 不因为缺少对端记录而丢失显示

### 2. 自动闭环失败兜底

如果某条记录在高置信判定中失败：

- 不自动丢弃
- 直接进入 `candidate`

### 3. 不可信候选绝不自动闭环

任何存在歧义的组，都必须停留在 `candidate`。

## 测试策略

### 后端

至少覆盖：

- 唯一 `OA ↔ 流水` 自动闭环
- 唯一 `流水 ↔ 发票` 自动闭环
- 唯一 `OA ↔ 流水 ↔ 发票` 自动闭环
- 一个 OA 多个流水 -> `candidate`
- 多个 OA 一笔流水 -> `candidate`
- 金额接近但不等 -> `candidate`
- 缺任一栏时仍能成组展示

### 前端

至少覆盖：

- `已配对` / `未配对` 按组渲染
- 同组记录横向对齐
- 某栏缺数据时单元格留空
- 某栏多条记录时垂直堆叠
- 详情按钮仍可打开记录详情
- 组布局与三栏拖拽兼容

## 交付结果

完成后，工作台的实际观察体验应变成：

- 财务先看“这一行是不是同一组候选”
- 再决定是否人工确认
- 而不是先分别在三张表里找记录再脑补关系

换句话说，本轮的目标不是“让系统更会猜”，而是“让系统只自动处理真正有把握的，并把剩下的候选关系排得足够直观”。
