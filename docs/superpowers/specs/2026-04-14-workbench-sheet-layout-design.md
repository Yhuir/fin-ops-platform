# 关联台三栏 Sheet 化样式重设计

日期：2026-04-14

## 目标

把 `已配对 / 未配对` 两个区域下的 `OA / 银行流水 / 进销项发票` 三栏，从当前偏 `block / card stack` 的视觉，重设计为更接近 Excel / sheet 的样式：

- 同一候选组仍然保持“一组一行、三栏横向对齐”
- 组内多条记录不再显示成独立卡片，而是显示成更轻的分割线列表
- 当出现 `1 条 OA 对多条发票`、`1 条流水对多条发票` 等场景时，用户能沿同一水平带看清对应关系

同时严格保持：

- 不改后端接口
- 不改候选组数据结构
- 不改现有列定义
- 不改现有 tag、按钮、动作、数据文案

本轮目标是“只改视觉层表达方式”，不是重做工作台交互模型。

## 现状问题

当前每个 pane 内的记录都渲染为独立 `record card`，并带有：

- 白色块状背景
- 独立卡片边框
- 圆角容器
- card hover 反馈

这种样式在单条记录时还可接受，但在 `一对多 / 多对一` 候选组里会出现两个问题：

1. 横向对应感弱  
   用户更容易看到“三个栏里各有几个块”，而不是“这一整带是一组关系”。

2. 多条记录时视觉切割过重  
   例如 `1 条 OA + 3 条发票`，右侧会像三个互相独立的小卡片，左侧 OA 又像另一个单独 block，用户不容易快速理解“左边这一条对应右边这三条”。

因此需要把视觉重心从“卡片”切回“表格分割线 + 同组背景带”。

## 设计原则

### 1. 保持当前分组模型不变

`group` 仍然是工作台唯一的渲染单位：

- 一组 = 一行
- 一行内仍然是：
  - `OA`
  - `银行流水`
  - `进销项发票`

本轮不调整：

- `group_id`
- `oa_rows / bank_rows / invoice_rows`
- 现有 group 排序、筛选、搜索和时间排序逻辑

### 2. 从 card 视觉切换为 sheet 视觉

每个候选组从“三个带圆角卡片的 pane slot”改成“一条 sheet band”：

- 组内背景统一而很浅
- 组内列之间用细竖线分隔
- 同一栏内多条记录之间用细横线分隔
- 取消 record 级厚边框和块状阴影

用户第一眼看到的应该是“同一条带上的对应关系”，而不是“几个互相独立的小块”。

### 3. 允许视觉上的“合并单元格”

当某栏只有一条记录而另一栏有多条记录时，视觉上应该接近 Excel 的合并单元格：

- 单条记录的 pane item 撑满整组高度
- 多条记录的 pane item 按行堆叠
- 单条项本身不需要重复出现在多行中

这是一种视觉上的 row-span 效果，不要求真正使用 `<table rowspan>`。

### 4. 现有信息密度不下降

当前列和数据密度已经比较高，本轮不能因为追求“表格感”而牺牲信息：

- 列顺序保持不变
- 列内容保持不变
- tag 保持不变
- 行内操作按钮保持不变
- hover / 选中 / 搜索高亮保持可辨识

## 视觉方案

### 整体结构

每个候选组保留当前三栏横向布局，但样式改成：

- 候选组外层：
  - 很浅的组背景
  - 更小的组间距
  - 组间主要依赖底部分割线和少量留白，而不是大块圆角卡片
- 三个 pane slot：
  - 视觉上更像同一条表格带的三段
  - 统一高度
  - 中间只保留轻量竖向分隔

推荐视觉语言：

- `group background`: 很浅的暖灰或蓝灰
- `column separators`: 1px 低对比度分割线
- `row separators`: 1px 更轻的横线
- `hover`: 轻微提亮，不再做卡片浮起感

## 组级背景色

为了让用户一眼知道“这些项属于同一组”，每个 group 需要一个统一的底色层。

建议：

- 底色只作用于 group 容器，不直接覆盖 tag 配色
- 颜色必须很浅，避免抢掉行内状态色
- 不同 group 不需要多彩轮换；统一一套浅背景即可

原因：

- 颜色的职责是“把一组兜起来”
- 不是给不同组编码不同语义
- 否则会与 tag / 状态色冲突

## 组内分隔

### 列间分隔

三栏之间用统一细竖线分隔：

- 不再依赖每个 pane 的独立完整边框
- 视觉上更像一整张 sheet 被切成三列

### 行间分隔

同一 pane 内的多条记录之间使用细横线：

- 线要比当前 card 边框更轻
- 不额外加圆角和块背景
- 最后一条记录不再重复加底线

## 单条与多条记录的视觉规则

### 单条记录

如果某个 pane 内只有 1 条记录：

- 该记录占满 pane slot 高度
- 内容垂直居中或按现有列规则排列
- 左右两侧多条记录不影响这条单项的完整展示

这对应视觉上的“合并单元格”。

### 多条记录

如果某个 pane 内有多条记录：

- 仍按当前顺序垂直堆叠
- 每条记录只保留内容和轻量分割线
- 不再出现独立小卡片感

### 空栏

如果某个 pane 在当前 group 内没有记录：

- 保持当前“当前栏暂无候选”的语义
- 但空栏背景也应融入 group sheet，而不是像单独空卡片
- 空栏文案要更像表格空白态，不要比真实数据更抢眼

## 现有元素兼容要求

### 列和数据

必须原样保留：

- 表头文案
- 列宽逻辑
- 列顺序持久化
- 行内文本结构
- 数值、时间、摘要、对方名称等现有排布

### Tag

所有现有 tag 保持：

- 文案不改
- 颜色不改
- 出现位置不改

sheet 化样式只改变它们周围的容器背景、边框和分隔关系。

### 按钮与动作

保留：

- `详情`
- `确认关联`
- `标记异常`
- `取消配对`
- 其他现有行级动作

但动作区视觉需要从“卡片尾部按钮区”过渡为“表格行内动作区”：

- 继续占据原有 action column
- 去掉独立卡片底色突兀感
- hover / selected 时与整行背景一起变化

## 状态与高亮

### Hover

当前的 card hover 更像“悬浮卡片”。重设计后应改为：

- 该记录所在行轻微提亮
- action 列同步提亮
- 不做阴影浮起

### Search Highlight

搜索命中仍然需要非常明确，但高亮方式应从“卡片描边”过渡为“表格行强调”：

- 可以继续保留浅黄背景
- 但外层描边应更贴近整行或整组，而不是像独立卡片描边

### Selected / Related

选中态和 related 态保持现有语义，但样式需要兼容 sheet 视觉：

- 不要因为去掉 card 边框而丢失辨识度
- 选中时优先强化整行或整组的背景、边线
- 不引入新的颜色体系

## 技术实现边界

本轮优先使用现有 DOM 结构完成：

- `CandidateGroupGrid`
- `CandidateGroupCell`
- `WorkbenchRecordCard`
- `styles.css`

允许做的事情：

- 调整 class 组织
- 为 group / pane / row 增加有限的样式语义 class
- 轻量调整容器结构，便于实现“单条撑满、多条分行”

不建议做的事情：

- 把整个区域重写成真正的 `<table>`
- 改动后端 payload
- 重写列配置系统
- 修改现有 cell renderer 文案和业务逻辑

## 风险点

### 1. 单条项撑满高度时的垂直对齐

如果某一栏只有 1 条而另一栏有多条，单条项需要撑满高度，但内部内容仍要可读：

- 文本不能被强行垂直拉散
- action column 不能跑位

### 2. 搜索高亮与浅组背景冲突

如果 group 底色和 search highlight 都是浅色，可能导致命中不明显。实现时需要保证：

- 命中色显著高于 group 底色
- 不影响 tag 颜色

### 3. 横向滚动时表头和内容观感一致

当前表头、内容、底部滚动轨道已经被对齐锁住。sheet 化后不能因为只改内容区样式而破坏：

- 表头边线节奏
- 内容区行分隔节奏
- 底部滚动轨道对齐

## 验收标准

- `已配对 / 未配对` 两个区域都改成更接近 Excel / sheet 的风格
- 同一 group 一眼能看出是“同一条带”
- `1 条 OA 对多条发票` 时，左侧单项视觉上像合并单元格，不再像独立 block
- 现有列、tag、按钮、动作、搜索、筛选、排序、详情都不改语义
- hover / selected / search highlight / related 等状态在新视觉中仍清楚可辨
- 不引入新的后端接口或数据结构变更
