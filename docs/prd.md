# 银企核销产品需求文档（PRD）

## 1. 背景

当前财务对账依赖多个外部系统导出的数据：

- 税务系统导出的销项发票表
- 税务系统导出的进项发票表
- 各家网银导出的银行流水明细

问题在于：

- 标准场景无法稳定自动平账
- 异常款项需要人工比对，成本高且容易遗漏
- 少收款、溢收款、缺票、退款、内部抵扣等状态难以持续追踪
- 后续催款、催票、退款缺乏统一台账

## 2. 产品目标

系统需要同时实现以下目标：

- 自动处理标准的发票与流水匹配
- 为异常场景提供高效率人工核销工作台
- 对所有未闭环事项建立后续台账和提醒
- 形成可审计的财务处理链路
- 为未来 OA 接入和项目成本测算预留可扩展模型

## 3. 用户角色

### 财务专员

- 导入发票和流水数据
- 查看自动匹配结果
- 在工作台中进行人工核销
- 维护欠款、缺票、退款等跟进信息

### 财务负责人

- 查看核销进度和异常台账
- 配置规则和差额处理策略
- 审核高风险人工核销动作

### 后续扩展角色

- OA 审批人
- 项目经理
- 经营分析人员

## 4. 核心业务流程

1. 财务导入销项发票、进项发票、银行流水
2. 系统进行标准化与自动匹配
3. 对完全匹配数据自动核销
4. 对未匹配或部分匹配数据生成待处理事项
5. 财务通过“左发票、右流水”工作台进行人工核销
6. 对处理结果生成催款、催票、退款、预收、预付等后续台账
7. 后续跟进完成后再回写闭环状态

## 5. 功能范围

## 5.1 数据导入

系统应支持：

- 批量导入销项发票
- 批量导入进项发票
- 批量导入银行流水
- 导入批次管理
- 字段校验、格式校验、重复检测
- 错误明细导出

导入后的数据应统一标准化：

- 客商名称规范化
- 金额字段统一
- 日期字段统一
- 正负金额统一口径

### 5.1.1 导入幂等与重复防重

系统必须保证同一份发票表或流水表被重复导入时，不会产生重复业务数据或重复记账。

防重策略采用 **双重防重机制：唯一业务主键 + 数据指纹**。

#### 发票防重规则

- 纸质发票或普通电子发票：以 `发票代码 + 发票号码` 作为唯一业务主键
- 数电票：以 `20位发票号码` 作为唯一业务主键
- 当导入数据命中已存在发票主键时，系统默认判定为重复发票，不再重复入库
- 如果同一主键对应的发票状态发生变化，例如变为 `已作废` 或 `已红冲`，系统应执行 `状态更新`，而不是简单跳过

#### 银行流水防重规则

- 优先使用 `银行交易流水号 / 网银凭证号 / 企银交易序号` 作为唯一业务主键
- 当官方流水唯一号存在时，以该字段作为流水幂等判断依据
- 若系统检测到该唯一号已存在，则默认判定为重复流水，不再重复入库

#### 无官方唯一号时的兜底方案

对于缺失官方流水号的老旧银行模板或第三方支付导出明细，系统应生成 `数据指纹` 作为兜底防重依据。

建议指纹输入要素：

- 我方账号
- 对方账户名称
- 交易日期
- 收付方向
- 交易金额

指纹生成后：

- 若库内不存在相同指纹，则按新增数据处理
- 若库内存在相同指纹，则标记为 `疑似重复`
- 若业务上确实可能存在“同日、同客商、同金额、同方向”的多笔交易，则不能直接自动跳过，应进入人工确认

#### 导入预览与执行规则

系统在用户上传文件后，不应立即写库，而应先生成一份 `导入解析报告`，至少分为：

- `本次新增`
- `状态更新`
- `重复跳过`
- `疑似重复待确认`
- `异常数据`

只有在用户确认导入后，系统才真正写入数据库。

#### 审计与追溯要求

每一条导入记录都应记录：

- 来源批次
- 原始行号
- 唯一业务主键
- 数据指纹
- 导入判定结果：新增 / 更新 / 重复跳过 / 疑似重复 / 异常
- 判定原因

#### 业务原则

- `重复导入` 不等于 `直接报错`
- 系统应优先把结果分类清楚，再由用户确认执行
- `状态更新` 与 `重复跳过` 必须明确区分
- 防重规则必须是实现层、数据库层、导入交互层三层一致

## 5.2 自动匹配

系统应优先根据以下规则生成自动匹配结果：

- 同一客商
- 金额完全一致
- 收付款方向一致

自动匹配结果分为：

- 自动核销成功
- 建议匹配，待人工确认
- 无法匹配，进入人工工作台

## 5.3 人工核销工作台

工作台是核心页面，需支持：

- 左侧查看待核销发票列表
- 右侧查看待核销流水列表
- 按客商、金额、状态、日期过滤
- 勾选后执行核销操作
- 查看候选匹配建议
- 查看历史核销记录和备注

工作台支持的动作至少包括：

- 一对一核销
- 人工关联
- 确认核销
- 异常处理
- 补充线下核销记录
- 录入内部抵扣单

### 5.3.1 工作台异常处理入口

当左侧与右侧都选中后，工作台右侧应出现 `异常处理` 区，而不是使用泛化的“报异常”按钮。

要求：

- 当前处于 `销项应收` 模式时，只展示销项异常选项
- 当前处于 `进项应付` 模式时，只展示进项异常选项
- 异常处理必须是结构化选项，不允许只靠自由文本备注
- 用户选择异常类型后，系统应决定是：
  - 生成提醒台账
  - 进入人工关联
  - 生成往来登记
  - 标记为待开票 / 待付款 / 追票 / 退款等后续动作

## 5.4 销项场景

### 正常场景

- 同一公司，发票金额等于收款金额
- 结果：已核销

### 少付款

- 发票金额大于收款金额
- 结果：部分核销
- 需要录入预计尾款收款日
- 自动进入催款台账

### 多票一付

- 同一公司多张发票金额之和等于一笔或多笔收款
- 支持人工合并核销

### 无票收款

- 存在收款流水但无对应发票
- 可标记为预收账款、押金、员工还款等
- 若需退款，则需要计划退款日

### 溢收款

- 收款金额大于发票金额
- 先核销对应发票，再拆分剩余金额为预收/往来待确认事项

### 5.4.1 销项发票异常处理标准选项

以下分类用于替代工作台中笼统的“报异常”入口：

- `SO-A 多票对应一个收入`：有收入，钱比销项少，多票对应一个收入，人工关联
- `SO-B 待收款`：有收入，钱比销项少，待收款，进入催收款提醒
- `SO-C 退款提醒`：有收入，钱比销项多，甲方多付，进入退款提醒
- `SO-D 一票对应多个收入`：有收入，钱比销项多，一票对应多个收入，人工关联
- `SO-E 待开销项票`：有收入，没有销项票，进入待开销项票
- `SO-F 外部往来-借入`：有收入，没有销项票，外部往来，别人借入，登记公司欠款
- `SO-G 外部往来-还款`：有收入，没有销项票，外部往来，别人还款，冲抵之前借给别人的记录
- `SO-H 非税收入`：有收入，没有销项票，不开票，登记为非税收入

## 5.5 进项场景

### 正常场景

- 同一公司，发票金额等于付款金额
- 结果：已核销

### 多付款 / 缺票

- 付款金额大于发票金额
- 结果：发票核销，剩余流水进入缺票台账

### 少付款 / 组合支付

- 发票金额大于公账付款金额
- 支持录入线下支付、个人垫付等补充核销记录
- 补录后允许强制平账

### 一付多票

- 一笔付款对应多张进项发票
- 支持人工合并核销

### 5.5.1 进项发票异常处理标准选项

以下分类用于工作台 `进项应付` 模式下的结构化异常处理：

- `PI-A 多张进项票合并付款`：有付款，钱比进项票多，多张进项票合并付款，人工关联
- `PI-B 追票`：有付款，钱比进项票多，进入追票
- `PI-C 无票追票`：有付款，无进项票，进入追票
- `PI-D 外部往来-借给别人`：有付款，无进项票，外部往来，借给别人，登记借出记录
- `PI-E 外部往来-归还借入`：有付款，无进项票，外部往来，还别人借入，关联公司还款
- `PI-F 待付款`：有付款，进项票比钱多，登记待付款并提醒
- `PI-G 部分现金支付`：有付款，进项票比钱多，部分现金支付
- `PI-H 多张支付合并`：有付款，进项票比钱多，多张支付合并，人工关联

## 5.6 全局高阶场景

### 差额核销

- 适用于手续费、抹零
- 差额需归类为财务费用、手续费或抹零

### 红字发票与退款

- 支持负数发票
- 支持与反向银行流水核销

### 债务互抵

- 同一客商同时是客户和供应商
- 支持跨板块内部抵扣，不依赖真实银行流水

## 5.7 台账与提醒

系统应自动生成并持续维护以下台账：

- 催款台账
- 催票台账
- 退款台账
- 预收款台账
- 预付款台账
- 待开销项票台账
- 待付款提醒台账
- 外部往来登记
- 非税收入登记

每条台账至少应包含：

- 来源单据
- 当前责任客商
- 未闭环金额
- 原因分类
- 预计处理日期
- 当前状态
- 跟进备注

## 5.8 审计与追溯

所有关键动作都必须记录：

- 操作人
- 操作时间
- 动作类型
- 关联单据
- 调整前后金额
- 备注和依据

## 6. 非功能要求

- 导入失败可定位到具体行
- 批量数据处理不能重复入账
- 重复导入必须具备幂等性
- 所有金额计算必须可追踪
- 人工核销动作可审计
- 状态变更可回溯
- 支持后续集成 OA 和项目维度

## 7. MVP 范围建议

第一版建议先交付以下内容：

- 三类数据导入
- 自动匹配
- 人工核销工作台
- 催款/催票/退款台账
- 处理 A/B/C/D/E/F/G/H/I/J 场景
- 审计日志

不纳入 MVP 的内容：

- 完整组织权限系统
- 复杂消息通知中心
- 项目损益报表
- OA 双向实时联动

## 8. 成功标准

- 标准一对一场景自动核销率显著提升
- 财务人工处理异常场景的时间显著下降
- 未闭环事项不再依赖线下表格跟踪
- 每条差异款项都能落到可跟进状态
- 后续能够无缝增加 OA 和项目维度
