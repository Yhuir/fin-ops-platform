## 财务“银企对账与发票核销”系统需求逻辑文档

### 一、 系统定位与核心目标

本系统旨在通过对比**业务单据（OA / 税务发票）与资金流水（网银记录）**，实现应收/应付账款的自动化与半自动化核销，彻底解决财务手工对账效率低、异常款项（少付、溢收、缺票等）难追踪的痛点，并建立完善的账期提醒机制。

### 二、 整体操作主流程

1. **数据接入 / 导入**：系统实时接入 OA 系统（或接入 MongoDB 中间库）同步审批单数据；财务定期导入外部税务系统下载的《销项发票表》和《进项发票表》，以及从各家网银导出的《银行流水明细》。
2. **系统初筛**：系统根据“对方公司名称”与“金额”进行自动匹配，将完全对应的数据自动平账。
3. **人工干预（核心）**：提供“左发票、右流水”的双屏对比界面，由财务人员手动处理未匹配的异常数据。
4. **闭环跟进**：对于处理后产生的新状态（如欠款、欠票），系统自动生成台账与代办提醒。

---

### 三、 销项板块逻辑（应收账款管理：我司开票，资金流入）

**核心原则**：以“收款流水”为核销基准，寻找对应的“销项发票”。

- **正常场景 (1对1完美匹配)**
    - **逻辑**：同一公司，发票金额 = 收款金额。
    - **处理**：状态变更为【已核销】，业务闭环。
- **异常A：少付款 (部分核销)**
    - **逻辑**：发票金额 > 收款金额（例：开票50，实收20）。
    - **处理**：系统核销20元，发票剩余待收30元，状态变更为【部分核销】。
    - **后续**：强制要求设置“预计尾款收款日”，进入【催款台账】。
- **异常B：多票一付 (合并核销)**
    - **逻辑**：同一公司多张发票金额之和 = 一笔/多笔收款金额（例：开票25+30，实收55）。
    - **处理**：人工勾选多张发票与对应流水，点击“合并核销”，状态变更为【已核销】。
- **异常C：无票收款 (预收款 / 往来款)**
    - **逻辑**：存在收款流水，但无对应销项发票（例：收到100元，未开票）。
    - **处理**：人工将流水标记为特定性质（如：预收账款、押金、员工还款等）。若为需退还的款项，需设置“计划退款日”进入提醒；若为预收，则留存等待未来开票后抵扣。
- **异常F：溢收款 (超额核销)**
    - **逻辑**：收款金额 > 发票金额（例：开票30，实收50）。
    - **处理**：优先核销该发票30元（状态变更为【已核销】），剩余20元流水系统自动剥离，转化为上述的**异常C（预收/往来款）**由人工确认。

---

### 四、 进项板块逻辑（应付账款管理：别人开票，资金流出）

**核心原则**：以“付款流水”为核销基准，寻找对应的“进项发票”。

- **正常场景 (1对1完美匹配)**
    - **逻辑**：同一公司，发票金额 = 付款金额。
    - **处理**：状态变更为【已核销】，业务闭环。
- **异常D：多付款 / 缺票 (预付账款)**
    - **逻辑**：付款金额 > 发票金额（例：开票5，实付10）。
    - **处理**：核销5元发票，流水剩余5元未平账，标记为【缺票】。
    - **后续**：进入【催票台账】，跟进对方补开发票。
- **异常E：少付款 / 组合支付**
    - **逻辑**：发票金额 > 公账付款金额（例：开票10，公账仅付8）。
    - **处理**：人工介入，补充“线下核销记录”（如备注：剩余2元已通过现金支付/个人垫付等）。补充后，发票强制平账，状态变更为【已核销】。
- **异常G：一付多票 (合并核销)**
    - **逻辑**：一笔付款金额 = 同一公司多张进项发票金额之和。
    - **处理**：人工勾选单笔流水与多张发票，进行“合并核销”。

---

### 五、 全局通用与高阶逻辑

这部分处理现实财务中必然遇到的“非标准化”阻力。

- **异常H：银行手续费与抹零**
    - **逻辑**：因银行扣费或零头抹除，导致流水与发票存在微小差额（例：发票100，收款99）。
    - **处理**：提供【差额核销】功能。人工确认后，将差额（1元）标记为“财务费用/手续费/抹零”。发票视为【已核销】，不再产生催款提醒。
- **异常I：红字发票与退款 (负数冲销)**
    - **逻辑**：发生退货，税务开具负数发票（如-10元），同时网银产生反向流水（如付款退给客户10元）。
    - **处理**：系统必须支持负数金额的核销抵扣，允许销项界面的负数发票与付款流水进行平账。
- **异常J：债务互抵 (客商身份重叠)**
    - **逻辑**：某公司既是客户也是供应商，双方同意货款互抵（例：欠我司50，我司欠他30，最终只收回20）。
    - **处理**：支持“跨板块平账”。财务可通过人工录入内部抵扣单，将30元的销项发票与30元的进项发票直接内部核销，不关联实际银行流水。

### 六、关联工作台与税金抵扣新增需求（2026-03-25）

本章节用于补充 UI 工作台、字段展示层次以及税金抵扣入口的产品需求。

#### 6.1 主工作台入口与总体结构

- 主页面名称为：`OA & 银行流水 & 进销项发票关联台`。
- 页面继续采用**上下两栏**结构：
  - 上半区：`已配对`
  - 下半区：`未配对`
- 页面继续采用**三栏并列**结构：
  - 左栏：OA
  - 中栏：银行流水
  - 右栏：进项 / 销项发票
- 页面风格应保持**Excel / 子表式工作台**，以表格、表头、行状态、行内操作为主，不采用大块卡片式布局。

#### 6.2 税金抵扣功能入口

- 在主工作台右上角新增按钮：`税金抵扣`。
- 点击后进入一个**独立页面**：`销项票税金 - 进项票税金`。
- 该页面用于按月份查看与处理：
  - 销项票税额
  - 进项票税额
  - 可抵扣税额
  - 抵扣后应纳 / 留抵结果
- 税金抵扣页面默认按所选月份加载相应销项票、进项票与税额汇总。

#### 6.3 月份选择

- 主工作台顶部新增一个**日历 / 月份选择按钮**。
- 用户选择月份后，系统应按该月份刷新显示：
  - OA 数据
  - 银行流水
  - 进项发票
  - 销项发票
- 税金抵扣页面也应继承同一月份上下文，或允许在页面内再次切换月份。

#### 6.4 三栏拖拽与动态显示

- 三栏之间的两个竖向分隔线必须支持**拖拽调整宽度**。
- 拖拽后，三栏宽度需同步作用于：
  - 上半区“已配对”
  - 下半区“未配对”
- 分隔线需要支持**完整拖动**：
  - 可将某一栏拖至 0 宽或接近 0 宽，视为该栏被收起
  - 可从收起状态再次拖回，恢复该栏显示
- 页面应支持动态显示：
  - 一栏
  - 两栏
  - 三栏
- 当用户收缩某一栏到极小宽度时，可视为该栏被临时折叠；重新展开后恢复表格视图。
- 各栏内部允许独立滚动与横向滚动，保证字段较多时仍保持 Excel 式查看体验。

#### 6.5 OA 栏直接显示字段

OA 栏主表需要直接显示以下信息：

- 申请人
- 项目名称
- 申请类型
- 金额
- 对方户名
- 申请事由
- OA 和流水关联情况

#### 6.6 银行流水栏直接显示字段

银行流水栏主表需要直接显示以下信息：

- 交易时间
- 借方发生额（支取）
- 贷方发生额（收入）
- 对方户名
- 支付账户
  - 需根据账号自动识别属于哪家银行流水
- 和发票关联情况
  - 详细信息查看
  - 关联情况
  - 取消关联
  - 异常处理
- 支付 / 收款时间
- 备注
  - 用于承接应收、应付、欠款、借款统计信息
- 还借款日期

#### 6.7 银行流水栏详情字段

银行流水栏在点击“查看详情”或点击行后，需在详情区域 / 抽屉中显示以下信息：

- 账号
- 账户名称
- 余额
- 币种
- 对方账号
- 对方开户机构
- 记账日期
- 摘要
- 备注
- 账户明细编号 / 交易流水号
- 企业流水号
- 凭证种类
- 凭证号

#### 6.8 发票栏直接显示字段

发票栏主表需要直接显示以下信息：

- 销方识别号
- 销方名称
- 购方识别号
- 购买方名称
- 开票日期
- 金额
- 税率
- 税额
- 价税合计
- 发票类型

#### 6.9 发票栏详情字段

发票栏在点击“查看详情”或点击行后，需在详情区域 / 抽屉中显示以下信息：

- 序号
- 发票代码
- 发票号码
- 数电发票号码
- 税收分类编码
- 特定业务类型
- 货物或应税劳务名称
- 规格型号
- 单位
- 数量
- 单价
- 发票来源
- 发票票种
- 发票状态
- 是否正数发票
- 发票风险等级
- 开票人
- 备注

#### 6.10 详情展示方式

- 所有“点击查看详情才显示的信息”不在主表内直接展开。
- 统一采用**右侧详情抽屉**方式展示。
- 点击表格中的某一行时，仅执行**选中**动作：
  - 当前行高亮
  - 同一组候选记录联动高亮
  - 不自动弹出详情抽屉
- 详情抽屉必须通过每一行内的 `详情` 按钮打开。
- 详情抽屉需支持根据当前选中行自动切换展示内容：
  - OA 详情
  - 银行流水详情
  - 发票详情

#### 6.11 行内操作要求

- 工作台中的主要动作应优先采用**行内操作**，避免单独底部操作条。
- 典型行内操作包括：
  - 确认关联
  - 标记异常
  - 取消关联
  - 查看详情
  - 修改异常类型
- 当前选中行可显示完整操作按钮，未选中行仅显示状态标签或下拉入口，以保证表格密度。

### 七、成本统计页面新增需求（2026-04-01）

本章节用于补充“成本统计”页面的业务需求。该页面建立在现有 `OA ↔ 银行流水 ↔ 发票` 工作台与项目归集底座之上，但定位是**统计与下钻页面**，不是新的核销弹窗。

#### 7.1 页面入口与位置

- 在顶部导航中，`税金抵扣` 右侧新增按钮：`成本统计`。
- 点击后进入一个**独立页面**，不是弹出窗口。
- 页面默认继承当前全局 `年份 / 月份` 上下文，也允许用户在页面内继续点击切换年月。

#### 7.2 成本统计数据来源

- 成本统计以**银行流水**为主数据源。
- 仅统计能够对应到 OA 字段的**支出类流水**。
- 每条进入成本统计的数据，至少需要能够取得以下 OA 字段：
  - 项目名称
  - 费用类型
  - 费用内容
- 如果一条流水尚未关联到 OA，或者 OA 中缺少上述关键字段，则该流水**不进入正式成本统计结果**。
- 收款流水、未确认关联流水、仅发票无流水的记录，不进入成本统计主视图。

#### 7.3 页面主视图一：按月份统计

- 页面默认打开 `按月份统计` 视图。
- 统计维度为：**当前所选月份**。
- 主表需要显示：
  - 项目名称
  - 费用类型
  - 金额
  - 费用内容
- 统计口径：
  - 在当前月份内，对已确认纳入成本统计的支出流水进行汇总展示。
  - 一行代表一组可解释的成本统计结果；允许按 `项目名称 + 费用类型 + 费用内容` 组合展示。
- 每一行均可点击继续下钻。

#### 7.4 页面主视图二：按项目名称统计

- 页面需支持按 `项目名称` 查看成本。
- 用户可从 `按月份统计` 中点击某一项目下钻进入，也可直接切换到 `按项目统计` 视图。
- 该视图需要显示：
  - 时间
  - 费用类别 / 费用类型
  - 金额
  - 费用内容
- 该视图仍然处于当前月份范围内，除非后续单独扩展跨月统计。

#### 7.5 层层下钻要求

- 成本统计页面中的每一条都应支持继续点击下钻。
- 推荐下钻链路为：
  1. 月份统计汇总
  2. 项目名称维度明细
  3. 具体银行流水明细
- 最终下钻到某一条具体流水时，页面需要能看到：
  - 该条银行流水本身信息
  - 对应 OA 的项目名称
  - 对应 OA 的费用类型
  - 对应 OA 的费用内容
- 下钻过程应保持在同一成本统计页面内，通过面包屑、返回按钮或层级切换完成，不使用弹窗承载主流程。

#### 7.6 导出要求

- 成本统计页面必须提供 `导出` 功能。
- 导出应至少支持导出**当前视图**的数据：
  - 当前月份汇总视图
  - 当前项目明细视图
  - 当前下钻到的流水明细视图
- 导出文件名应带有清晰上下文，例如：
  - 年月
  - 当前视图层级
  - 项目名称（如适用）

#### 7.7 页面排版要求

- 页面排版必须清晰，不可做成拥挤的纯技术报表。
- 推荐保留：
  - 年月选择
  - 统计视图切换
  - 面包屑 / 返回路径
  - 汇总信息区
  - 主表区
  - 导出按钮
- 主表仍以清晰的表格为主，保证财务人员可以快速：
  - 看项目
  - 看费用类型
  - 看金额
  - 看费用内容
  - 继续下钻

#### 7.8 OA 费用类型字典

成本统计页面中，`费用类型` 采用以下 OA 字典值：

- 设备货款及材料费
- 人工费/劳务费/服务费
- 住宿费
- 招待费（餐费、烟酒等）
- 交通费
- 车辆使用费（汽油、过路、保险、维修、税费等）车辆维修
- 运费/邮费/杂费
- 房屋使用费（户租、水电、维修、车位、屋业等）
- 经营/办公费用
- 财务费用
- 借款
- 还款
- 其他
- 固定资产

#### 7.9 与现有页面的关系

- `关联台` 负责核销、异常处理、忽略、已处理异常等流程。
- `税金抵扣` 负责进销项税额抵扣。
- `成本统计` 负责基于已纳入统计口径的银行流水做项目与费用视角统计。
- 三个页面共用同一套年份 / 月份上下文，但职责不同：
  - 关联台：处理关系
  - 税金抵扣：处理税额
  - 成本统计：处理费用统计与下钻分析
